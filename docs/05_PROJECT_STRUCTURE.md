# Project Structure & Modularity Guide
## Mobile Shop — Inventory, Billing & Sales Analytics System

**Version:** 1.0 (Draft for Review)
**Architecture:** Modular, Domain-Driven Next.js Monorepo

---

## 1. Architecture Philosophy

This project is organized around **business domains**, not technical layers. Each domain (Auth, Products, Bills, Analytics) owns its API routes, database models, components, and types together. This makes the codebase:

- **Self-contained** — Want to add a new feature to Bills? Everything is in one place.
- **Easy to test** — Mock a domain, test in isolation.
- **Scalable** — Can split into microservices later if needed (each domain becomes a separate service).
- **Maintainable** — New developers understand "Bills owns billing logic" without hunting across files.

## 3. Domain Breakdown

### **Auth Domain**

- **Types:** `SessionUser`, `AuthContext`

- **Types:** `Product`, `ProductUnit`, `StockInRecord`
- **Hooks:** `useProducts`, `useStockIn`, `useCategories`
- **Database Models:** `Product`, `ProductUnit`, `StockInRecord`, `Category`

### **Bills Domain**
- **Routes:** `/api/v1/bills`, `/api/v1/bills/[id]`, `/api/v1/bills/[id]/void`
- **Pages:** `/billing`, `/billing/cart`, `/billing/history`
- **Components:** `ProductSearch`, `BillCart`, `BillItem`, `BillPreview`, `CustomerInfoForm`, `BillPrintable`
- **Types:** `Bill`, `BillItem`
- **Hooks:** `useBills`
- **Database Models:** `Bill`, `BillItem`, `Customer`
- **Transactions:** Stock deduction logic in `lib/db-transactions.ts`

### **Analytics Domain**
- **Routes:** `/api/v1/analytics/monthly`
- **Pages:** `/analytics`
- **Components:** `RevenueChart`, `CategoryBreakdown`, `TopProducts`, `MonthSelector`, `ComparisonCard`
- **Types:** `AnalyticsResponse`
- **Hooks:** `useAnalytics`
- **Database Queries:** Direct Prisma aggregations (no separate analytics table in v1)

### **Users Domain**
- **Routes:** `/api/v1/users`
- **Pages:** `/staff-management`
- **Components:** Staff list + create/edit forms
- **Types:** `User` (subset visible to owner only)
- **Database Models:** `User`

## 4. Key Architectural Decisions

### **4.1 Why API Routes in `/api/v1/`?**
- Keeps frontend and backend in one codebase → single deployment to Vercel
- Same TypeScript codebase → share types between frontend and backend
- No separate server process in development — just `npm run dev`
- All endpoints documented via OpenAPI if needed later

### **4.2 Why Prisma?**
- **Type Safety:** Prisma generates TypeScript types from schema automatically
- **Atomic Transactions:** Native support for multi-statement transactions (stock deduction + bill creation)
- **Migrations:** Versioned, traceable schema changes
- **Zero-Boilerplate:** No need to hand-write SQL queries

### **4.3 Why NextAuth.js?**
- **Built for Next.js:** Integrates seamlessly with API routes
- **JWT Sessions:** Stateless, role claims embedded in token
- **Callback Hooks:** Can enforce RBAC at the callback level before route handler runs
- **No Session DB:** Uses encrypted JWT by default (simpler than managing a session table)

### **4.4 Why Atomic Transactions for Bills?**
- A bill is ONLY created if **all** stock checks pass and **all** inventory decrements succeed in the same transaction
- If the transaction fails partway (e.g. one item goes out of stock between check and deduction), the entire bill is rolled back
- Prevents the nightmare scenario where a bill exists but stock is wrong

**Example:**
```typescript
// In lib/db-transactions.ts
async function createBillWithStockDeduction(items, discount, customerId) {
  const result = await prisma.$transaction(async (tx) => {
    // 1. Check all items have sufficient stock
    for (const item of items) {
      const product = await tx.product.findUnique({ where: { id: item.productId } });
      if (product.quantity_in_stock < item.quantity) {
        throw new Error(`${product.name} out of stock`);
      }
    }
    
    // 2. If all checks pass, create bill and decrement stock
    const bill = await tx.bill.create({ data: { ... } });
    for (const item of items) {
      await tx.product.update({
        where: { id: item.productId },
        data: { quantity_in_stock: { decrement: item.quantity } }
      });
    }
    return bill;
  });
  return result;
}
```

### **4.5 Why Domain-Grouped Folder Structure?**
Instead of:
```
api/
  auth/
  products/
  bills/
components/
  auth/
  products/
  bills/
types/
  auth/
  products/
  bills/
```

We use **domain-first grouping** where API routes, components, and types for "Bills" live together. This:
- Makes it obvious where to add new bill features
- Makes it easy to find all code related to a domain
- Scales well if you later split a domain into a microservice

## 5. Development Workflow

### **To Add a New Feature (e.g., "Allow refunds"):**

1. **Update Prisma schema** (`prisma/schema.prisma`) — add a `Refund` model
2. **Run migration** — `npx prisma migrate dev --name add_refund_model`
3. **Create API route** — `app/api/v1/refunds/route.ts`
4. **Create components** — `components/billing/RefundForm.tsx`, `RefundList.tsx`
5. **Create hooks** — `hooks/useRefunds.ts`
6. **Create types** — `types/models.ts` (auto-updated by Prisma)
7. **Create page** — `app/(dashboard)/refunds/page.tsx`
8. **Write tests** — `__tests__/api/refunds.test.ts`, `__tests__/components/RefundForm.test.tsx`

Everything for the "refunds" feature is organized in one place. Add, test, deploy.

## 6. Modularity Benefits

| Benefit | How It's Achieved |
|---|---|
| **Easy to test** | Mock Prisma, test route handlers in isolation |
| **Easy to extend** | Add routes/components/types to a domain without touching others |
| **Easy to navigate** | Know exactly where bills code is, not scattered across files |
| **Easy to onboard** | New dev reads the folder structure and understands the system |
| **Easy to scale** | If Bills gets complex, extract it into a separate Next.js service behind a gateway |
| **Type safe** | TypeScript enforced across API routes, components, database queries |

## 7. Local Development Setup

```bash
# 1. Clone & install
npm install

# 2. Copy .env.example to .env.local, fill in DATABASE_URL
cp .env.example .env.local

# 3. Run migrations
npx prisma migrate dev

# 4. Optionally seed dummy data
npx prisma db seed

# 5. Start dev server
npm run dev

# 6. Open http://localhost:3000
```

That's it. No separate Python virtual environment, no separate backend server startup, no two databases to manage locally.

## 8. Testing Strategy

- **API routes:** Test stock deduction atomicity, RBAC checks, bill creation with concurrent requests
- **Components:** Test UI renders correctly, form validation works, error messages appear
- **Database:** Use a test database (or transactions that rollback) to avoid polluting dev data
- **E2E (future):** Playwright tests for full user workflows (login → add stock → bill → analytics)

See [PHASE_DISTRIBUTION.md](./06_PHASE_DISTRIBUTION.md) for testing roadmap by phase.
